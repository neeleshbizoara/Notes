## Q28. Explain the Redux data flow (action ŌåÆ reducer ŌåÆ store ŌåÆ view).

**Answer:**

Redux follows a **one-way data flow**:

```
User clicks button ŌåÆ Dispatch ACTION ŌåÆ REDUCER processes it ŌåÆ STORE updates ŌåÆ VIEW re-renders
```

**Step-by-step with banking example:**

```jsx
// 1. ACTION - What happened (plain object)
const depositAction = { type: 'account/deposit', payload: 5000 };

// 2. REDUCER - How state changes (pure function)
function accountReducer(state = { balance: 0 }, action) {
  switch (action.type) {
    case 'account/deposit':
      return { ...state, balance: state.balance + action.payload };
    case 'account/withdraw':
      return { ...state, balance: state.balance - action.payload };
    default:
      return state;
  }
}

// 3. STORE - Single source of truth
import { createStore } from 'redux';
const store = createStore(accountReducer);

// 4. VIEW - React component subscribes to store
function AccountBalance() {
  const balance = useSelector(state => state.balance);
  const dispatch = useDispatch();

  return (
    <div>
      <p>Balance: Ōé╣{balance}</p>
      <button onClick={() => dispatch({ type: 'account/deposit', payload: 5000 })}>
        Deposit Ōé╣5000
      </button>
    </div>
  );
}
```

## Q29. What is Redux Toolkit and how does it simplify traditional Redux?

**Answer:**

Redux Toolkit (RTK) is the **official recommended way** to write Redux. It eliminates boilerplate.

**Traditional Redux (lots of code):**
```jsx
// action types
const DEPOSIT = 'account/deposit';
const WITHDRAW = 'account/withdraw';

// action creators
function deposit(amount) { return { type: DEPOSIT, payload: amount }; }
function withdraw(amount) { return { type: WITHDRAW, payload: amount }; }

// reducer
function accountReducer(state = { balance: 0 }, action) {
  switch (action.type) {
    case DEPOSIT: return { ...state, balance: state.balance + action.payload };
    case WITHDRAW: return { ...state, balance: state.balance - action.payload };
    default: return state;
  }
}

// store
const store = createStore(accountReducer);
```

**Redux Toolkit (same thing, much less code):**
```jsx
import { createSlice, configureStore } from '@reduxjs/toolkit';

const accountSlice = createSlice({
  name: 'account',
  initialState: { balance: 0 },
  reducers: {
    deposit(state, action) {
      state.balance += action.payload;  // Can "mutate" directly (Immer handles it)
    },
    withdraw(state, action) {
      state.balance -= action.payload;
    }
  }
});

export const { deposit, withdraw } = accountSlice.actions;  // Auto-generated!
export default accountSlice.reducer;

const store = configureStore({
  reducer: { account: accountSlice.reducer }
  // Automatically adds Redux DevTools + middleware
});
```

**Key benefits of RTK:**
| Feature | Traditional Redux | Redux Toolkit |
|---|---|---|
| Action types | Manual strings | Auto-generated |
| Action creators | Write manually | Auto-generated by `createSlice` |
| Immutability | Spread operator everywhere | Write "mutations" (Immer does immutability) |
| Store setup | Manual middleware config | `configureStore` does it all |
| Async operations | Install thunk/saga separately | `createAsyncThunk` built-in |
| API calls | Write fetch + actions manually | `RTK Query` handles everything |

## Q30. Explain `createSlice`, `createAsyncThunk`, and `RTK Query`.

**Answer:**

**1. `createSlice` ŌĆö Defines state, reducers, and actions in one place:**
```jsx
const transactionSlice = createSlice({
  name: 'transactions',
  initialState: {
    list: [],
    filter: 'all'
  },
  reducers: {
    setFilter(state, action) {
      state.filter = action.payload;
    },
    addTransaction(state, action) {
      state.list.unshift(action.payload);
    }
  },
  extraReducers: (builder) => {
    // Handle async actions here (from createAsyncThunk)
  }
});
```

**2. `createAsyncThunk` ŌĆö Handles API calls with loading/error states:**
```jsx
const fetchTransactions = createAsyncThunk(
  'transactions/fetch',
  async (accountId, { rejectWithValue }) => {
    try {
      const response = await fetch(`/api/accounts/${accountId}/transactions`);
      if (!response.ok) throw new Error('Failed to fetch');
      return await response.json();
    } catch (err) {
      return rejectWithValue(err.message);
    }
  }
);

// In the slice:
extraReducers: (builder) => {
  builder
    .addCase(fetchTransactions.pending, (state) => {
      state.loading = true;
      state.error = null;
    })
    .addCase(fetchTransactions.fulfilled, (state, action) => {
      state.loading = false;
      state.list = action.payload;
    })
    .addCase(fetchTransactions.rejected, (state, action) => {
      state.loading = false;
      state.error = action.payload;
    });
}

// In component:
dispatch(fetchTransactions('ACC-12345'));
```

**3. `RTK Query` ŌĆö Complete data fetching and caching solution:**
```jsx
import { createApi, fetchBaseQuery } from '@reduxjs/toolkit/query/react';

const bankApi = createApi({
  reducerPath: 'bankApi',
  baseQuery: fetchBaseQuery({ baseUrl: '/api/' }),
  tagTypes: ['Account', 'Transaction'],
  endpoints: (builder) => ({
    // GET request
    getAccounts: builder.query({
      query: () => 'accounts',
      providesTags: ['Account']
    }),
    // GET with parameter
    getTransactions: builder.query({
      query: (accountId) => `accounts/${accountId}/transactions`,
      providesTags: ['Transaction']
    }),
    // POST request
    transferMoney: builder.mutation({
      query: (body) => ({ url: 'transfer', method: 'POST', body }),
      invalidatesTags: ['Account', 'Transaction']  // Auto-refetch after mutation!
    })
  })
});

export const { useGetAccountsQuery, useGetTransactionsQuery, useTransferMoneyMutation } = bankApi;

// Usage in component:
function AccountList() {
  const { data: accounts, isLoading, error } = useGetAccountsQuery();
  // That's it! Caching, loading states, error handling ŌĆö all automatic!
  
  if (isLoading) return <Spinner />;
  if (error) return <Error />;
  return accounts.map(acc => <AccountCard key={acc.id} {...acc} />);
}
```

## Q31. What is middleware in Redux? Compare Redux Thunk vs Redux Saga.

**Answer:**

**Middleware** sits between dispatching an action and the reducer. It can intercept, modify, delay, or cancel actions.

```
dispatch(action) ŌåÆ MIDDLEWARE ŌåÆ reducer
```

**Redux Thunk (Simple ŌĆö function-based):**
```jsx
// Thunk = action creator that returns a FUNCTION instead of an object
function transferMoney(fromAcc, toAcc, amount) {
  return async (dispatch, getState) => {
    dispatch({ type: 'transfer/start' });
    try {
      const result = await api.transfer(fromAcc, toAcc, amount);
      dispatch({ type: 'transfer/success', payload: result });
    } catch (err) {
      dispatch({ type: 'transfer/error', payload: err.message });
    }
  };
}

// Usage
dispatch(transferMoney('ACC-001', 'ACC-002', 5000));
```

**Redux Saga (Complex ŌĆö generator-based):**
```jsx
import { call, put, takeLatest } from 'redux-saga/effects';

function* transferMoneySaga(action) {
  try {
    yield put({ type: 'transfer/start' });
    const result = yield call(api.transfer, action.payload);
    yield put({ type: 'transfer/success', payload: result });
  } catch (err) {
    yield put({ type: 'transfer/error', payload: err.message });
  }
}

// Watch for the action
function* watchTransfer() {
  yield takeLatest('TRANSFER_REQUEST', transferMoneySaga);
}
```

**Comparison:**

| Feature | Thunk | Saga |
|---|---|---|
| Learning curve | Easy | Steep (generators) |
| Code complexity | Simple | More verbose |
| Testing | Harder (mock dispatch) | Easier (just test yielded values) |
| Complex flows | Basic async/await | Race conditions, cancellation, debounce |
| Use case | Simple API calls | Complex workflows (multi-step transactions) |
| Bundle size | Tiny (14 lines of code) | Larger |

**Interview tip:** "For most projects, Thunk (or createAsyncThunk) is sufficient. I'd use Saga only when I need complex orchestration ŌĆö like a multi-step payment flow where I need to handle cancellation, retries, and rollbacks."

## Q32. When would you use Context API vs Redux? What are the performance implications?

**Answer:**

| Feature | Context API | Redux |
|---|---|---|
| Best for | Low-frequency updates (theme, auth, language) | High-frequency updates (forms, lists, real-time data) |
| Setup | Minimal | More setup (but RTK simplifies it) |
| DevTools | No | Excellent (time-travel debugging) |
| Performance | ALL consumers re-render on any change | Only subscribed components re-render |
| Middleware | None | Thunk, Saga, logging, etc. |
| Bundle size | 0 KB (built into React) | ~10 KB (Redux + RTK) |

**Performance problem with Context:**
```jsx
// ŌØī Context: ALL consumers re-render when ANY value changes
const AppContext = createContext();

function AppProvider({ children }) {
  const [user, setUser] = useState(null);
  const [theme, setTheme] = useState('light');
  const [notifications, setNotifications] = useState([]);

  return (
    <AppContext.Provider value={{ user, theme, notifications, setTheme }}>
      {children}
    </AppContext.Provider>
  );
}

// When theme changes ŌåÆ UserProfile, NotificationBell, and Header ALL re-render!
// Even though they only use `user` or `notifications`
```

**Solution ŌĆö Split contexts:**
```jsx
const UserContext = createContext();
const ThemeContext = createContext();
const NotificationContext = createContext();
// Now changing theme doesn't re-render user-related components
```

**Redux doesn't have this problem:**
```jsx
// Only re-renders when `user` specifically changes
const user = useSelector(state => state.auth.user);

// Only re-renders when `theme` specifically changes
const theme = useSelector(state => state.ui.theme);
```

**My rule of thumb:**
- **Context:** Auth state, theme, language, feature flags (change rarely)
- **Redux:** Transaction data, form state, dashboard data, real-time prices (change often)

## Q34. What is Redux selector pattern? Explain `reselect` and memoized selectors.

**Answer:**

**Selectors** are functions that extract specific data from the Redux store. **Memoized selectors** cache results and only recalculate when inputs change.

```jsx
// Simple selector (runs every time)
const selectAllTransactions = (state) => state.transactions.list;
const selectFilter = (state) => state.transactions.filter;

// ŌØī This recalculates on EVERY re-render, even if nothing changed
function TransactionList() {
  const transactions = useSelector(state => 
    state.transactions.list.filter(t => t.type === state.transactions.filter)
  );
}

// Ō£ģ Memoized selector with reselect
import { createSelector } from '@reduxjs/toolkit';  // reselect is built into RTK

const selectFilteredTransactions = createSelector(
  [selectAllTransactions, selectFilter],  // Input selectors
  (transactions, filter) => {
    // This only runs when transactions or filter actually change
    console.log('Filtering...');  // Won't run unnecessarily
    if (filter === 'all') return transactions;
    return transactions.filter(t => t.type === filter);
  }
);

// Usage
function TransactionList() {
  const filteredTxns = useSelector(selectFilteredTransactions);
  // If other state changes (like UI), this does NOT recalculate Ō£ģ
}
```

**More complex selector:**
```jsx
const selectAccountSummary = createSelector(
  [selectAccounts, selectTransactions],
  (accounts, transactions) => {
    return accounts.map(acc => ({
      ...acc,
      recentTransactions: transactions
        .filter(t => t.accountId === acc.id)
        .slice(0, 5),
      totalSpent: transactions
        .filter(t => t.accountId === acc.id && t.type === 'debit')
        .reduce((sum, t) => sum + t.amount, 0)
    }));
  }
);
```

## Q35. How do you handle optimistic updates in Redux for a payment processing screen?

**Answer:**

**Optimistic update** = Update the UI immediately (assume success), then roll back if the API call fails.

```jsx
// Using RTK Query optimistic update
const bankApi = createApi({
  endpoints: (builder) => ({
    transferMoney: builder.mutation({
      query: (transfer) => ({
        url: 'transfer',
        method: 'POST',
        body: transfer
      }),
      // Optimistic update
      async onQueryStarted(transfer, { dispatch, queryFulfilled }) {
        // 1. Immediately update the UI (before API responds)
        const patchResult = dispatch(
          bankApi.util.updateQueryData('getAccounts', undefined, (draft) => {
            const fromAcc = draft.find(a => a.id === transfer.fromAccountId);
            const toAcc = draft.find(a => a.id === transfer.toAccountId);
            if (fromAcc) fromAcc.balance -= transfer.amount;
            if (toAcc) toAcc.balance += transfer.amount;
          })
        );

        try {
          // 2. Wait for API to confirm
          await queryFulfilled;
          // Success! UI already shows correct data
        } catch {
          // 3. API failed ŌĆö UNDO the optimistic update
          patchResult.undo();
          dispatch(showNotification({ type: 'error', message: 'Transfer failed! Reverting...' }));
        }
      }
    })
  })
});
```

**Without RTK Query (manual approach):**
```jsx
function transferMoney(fromAcc, toAcc, amount) {
  return async (dispatch, getState) => {
    // Save current state for rollback
    const previousState = getState().accounts;
    
    // Optimistic update
    dispatch(updateBalance({ accountId: fromAcc, amount: -amount }));
    dispatch(updateBalance({ accountId: toAcc, amount: +amount }));
    dispatch(addTransaction({ status: 'pending', amount }));

    try {
      const result = await api.transfer(fromAcc, toAcc, amount);
      dispatch(confirmTransaction(result.transactionId));
    } catch (error) {
      // Rollback
      dispatch(restoreAccounts(previousState));
      dispatch(removeTransaction());
      dispatch(showError('Transfer failed. Please try again.'));
    }
  };
}
```

## Q36. What is the difference between `redux-persist` and manual localStorage management?

**Answer:**

| Feature | `redux-persist` | Manual `localStorage` |
|---|---|---|
| Setup | Library with config | Write your own code |
| Selective persistence | Whitelist/blacklist slices | Manual for each field |
| Rehydration | Automatic on app load | Manual loading logic |
| Storage engines | localStorage, sessionStorage, AsyncStorage, etc. | You choose and implement |
| Migration | Built-in state migration | Build your own |
| Transforms | Encrypt, compress, filter | Do it yourself |

```jsx
// redux-persist setup
import { persistStore, persistReducer } from 'redux-persist';
import storage from 'redux-persist/lib/storage';

const persistConfig = {
  key: 'root',
  storage,
  whitelist: ['auth', 'preferences'],  // Only save these slices
  blacklist: ['transactions', 'ui']     // Don't save these
};

const persistedReducer = persistReducer(persistConfig, rootReducer);
const store = configureStore({ reducer: persistedReducer });
const persistor = persistStore(store);

// In App.jsx
import { PersistGate } from 'redux-persist/integration/react';

function App() {
  return (
    <Provider store={store}>
      <PersistGate loading={<Splash />} persistor={persistor}>
        <BankingApp />
      </PersistGate>
    </Provider>
  );
}
```

**Important for financial apps:**
- NEVER persist sensitive data (tokens, PINs, account numbers) in localStorage
- Use `sessionStorage` for session-only data
- Use `transforms` to encrypt what you persist

## Q37. Explain normalized state shape. Why is it important for a transaction history module?

**Answer:**

**Normalized** = Store data like a database. Each entity stored by ID, with relationships as IDs (not nested objects).

```jsx
// ŌØī Denormalized (nested) - common mistake
const state = {
  accounts: [
    {
      id: 'ACC-001',
      name: 'Savings',
      transactions: [                  // Nested!
        { id: 'TXN-1', amount: 5000, description: 'Salary' },
        { id: 'TXN-2', amount: -200, description: 'UPI' },
      ]
    },
    {
      id: 'ACC-002',
      name: 'Current',
      transactions: [
        { id: 'TXN-3', amount: -1000, description: 'Rent' },
      ]
    }
  ]
};
// Problem: To update TXN-2, you need to: find the account ŌåÆ find the transaction ŌåÆ update
// Very complex and error-prone!

// Ō£ģ Normalized (flat) - like a database
const state = {
  accounts: {
    byId: {
      'ACC-001': { id: 'ACC-001', name: 'Savings', transactionIds: ['TXN-1', 'TXN-2'] },
      'ACC-002': { id: 'ACC-002', name: 'Current', transactionIds: ['TXN-3'] }
    },
    allIds: ['ACC-001', 'ACC-002']
  },
  transactions: {
    byId: {
      'TXN-1': { id: 'TXN-1', amount: 5000, description: 'Salary', accountId: 'ACC-001' },
      'TXN-2': { id: 'TXN-2', amount: -200, description: 'UPI', accountId: 'ACC-001' },
      'TXN-3': { id: 'TXN-3', amount: -1000, description: 'Rent', accountId: 'ACC-002' }
    },
    allIds: ['TXN-1', 'TXN-2', 'TXN-3']
  }
};

// Now updating TXN-2 is simple:
state.transactions.byId['TXN-2'].amount = -250;  // Direct access by ID!
```
